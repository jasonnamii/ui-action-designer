# PRD Frames

PRD 유형 분류 + 유형별 Action·Task 매핑 + PRD 작성 모드.

---

## 1. PRD 유형 분류기

| # | 유형 | 구조 특징 | 판별 키워드 |
|---|------|----------|-----------|
| 1 | **기능명세형** | 기능 목록 + 스펙 테이블 | "기능 요구사항", "FR-", "기능 정의" |
| 2 | **유저스토리형** | As a~, I want~, So that~ | "As a", "User Story", "AC" |
| 3 | **JTBD형** | Job Statement + Forces + Outcome | "Job", "When I~", "Hire/Fire", "Forces" |
| 4 | **플로우형** | 유저 플로우 + 분기 조건 | "Flow", "분기", "시퀀스", "→" |
| 5 | **화면명세형** | 화면별 요소·동작 정의서 | "화면 ID", "Screen", "요소", "동작 정의" |

**판별:** 키워드 2+→확정 / 1→구조 보충 판별 / 복합형→주+보조 병기, 주 유형 룰 우선.
**보조 유형 활용:** Task 순서·의존성 보충 + SHE 입력 보충만. 보조 유형 매핑룰로 Action·Task 추출 ✗

## 2. 유형별 매핑 룰

### 2-1. 기능명세형

| PRD 요소 | → 매핑 | 규칙 |
|----------|--------|------|
| 기능명/ID | Action | 1기능=1Action. 하위 기능=별도 Action |
| 상세 요구사항 | Task | 각 항목=1Task |
| 선행 조건 | Task 의존성 | 순서 제약 |
| 비기능 요구사항 | 제외 | SHE 참고 입력만 |

### 2-2. 유저스토리형

| PRD 요소 | → 매핑 | 규칙 |
|----------|--------|------|
| want절 | Action | "I want to [동사구]"=Action명 |
| So that절 | Action KPI 힌트 | 목적→전환 지표 |
| AC 항목 | Task | 각 AC=1Task |
| Epic/Theme | Action 그룹 | 복수 Story→Action 간 흐름 |

### 2-3. JTBD형

| PRD 요소 | → 매핑 | 규칙 |
|----------|--------|------|
| Job Statement want절 | Action | "I want to~"=Action |
| Desired Outcome | Action KPI | 성공 기준→측정 지표 |
| Forces (Push/Pull) | Task 우선순위 | 강한 항목→핵심 경로 Task |
| Switching Trigger | 진입 Task | 전환 계기→첫 Task 설계 |

### 2-4. 플로우형

| PRD 요소 | → 매핑 | 규칙 |
|----------|--------|------|
| 종착점 | Action | 각 End State=1Action 완료 |
| 분기 노드 | Task | 각 분기 판단=1Task |
| 선형 스텝 | Task | 순차 Task |
| 예외/에러 분기 | 보조 Task | SHE H단계 숨김 후보 |

### 2-5. 화면명세형

| PRD 요소 | → 매핑 | 규칙 |
|----------|--------|------|
| 화면 목적/타이틀 | Action | 1화면=1Action (복합→분리) |
| UI 요소별 동작 | Task | 각 인터랙션=1Task |
| 화면 전환 조건 | Task 순서 | 트리거→의존성 |
| 상태별 변형 | SHE 입력 | H단계 트리거 후보 |

## 3. PRD 작성 모드

조건: PRD 미제공 + 형 "PRD부터" 지시. Action·Task 특성에 맞는 팩터만 선택→심플 PRD.

### 풀 프레임 (12팩터)

| # | 팩터 | 선택 기준 |
|---|------|----------|
| 1 | 목적 정의 | 항상 |
| 2 | 타겟 사용자 | 항상 |
| 3 | 핵심 Action | 항상 |
| 4 | Task 분해 | 항상 |
| 5 | 성공 지표 | 전환·퍼널 관련 |
| 6 | 진입 조건 | 딥링크·조건부 진입 |
| 7 | 예외 흐름 | 오류 비용 높은 Action |
| 8 | 상태 분기 | 상태별 화면 변형 |
| 9 | 데이터 요구사항 | 외부 데이터 의존 |
| 10 | 비기능 요구사항 | 규제·대용량·실시간 |
| 11 | 의존성 | 멀티스텝 플로우 |
| 12 | 우선순위 | 스코프 판단 필요 시 |

### 선택 규칙

| Action 특성 | 필수 | 선택 | 제외 |
|-------------|------|------|------|
| 단일+선형 (검색, 조회) | 1,2,3,4 | 5 | 6~12 |
| 전환 (구매, 결제) | 1,2,3,4,5,7 | 6,8 | 9~12 |
| 멀티스텝 (온보딩, 예약) | 1,2,3,4,5,6,11 | 7,8 | 9,10,12 |
| 조건부 분기 (대시보드) | 1,2,3,4,8 | 5,6 | 7,9~12 |
| 데이터 중심 (피드, 지도) | 1,2,3,4,9 | 5,6 | 7,8,10~12 |
| 규제·보안 (인증, 금융) | 1,2,3,4,5,7,10 | 6,8,9 | 11,12 |

**산출:** 유저스토리형 PRD 통일. `## [Action명]` → 목적·타겟·(성공지표) → Tasks 테이블(#·Task·AC·비고) → 해당 선택팩터만.
✗ 선택 안 된 팩터 빈 섹션 / ✗ 12개 전부 나열 / ✓ 30초 이내 분량
**A경로 복귀:** 생성 PRD=유저스토리형→§2-2 매핑룰 적용.

## 4. 정합성 검증

⑧산출에서 PRD vs UI 제안 커버리지 검증.

**매트릭스:** PRD Task(AC 요약) × 4제안(표준/iOS26/AOS/창의적) → ✓(충족)/△(부분)/✗(미충족)+사유

**규칙:** 핵심경로 Task ✗→수정재제안 / △2+→의도적 생략 확인 / ✓80%미만→스코프 괴리 분석
